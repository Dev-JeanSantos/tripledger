# ADR-002 — Design da API REST

> [Versão em inglês](../en/ADR-002.md)

## Status
Accepted

## Data
2026-02-22

## Contexto

O TripLedger irá expor funcionalidades via APIs REST. O sistema deve:

- Suportar operações CRUD para Viagens, Despesas, Itens de Checklist e Usuários
- Ser modular e manter separação rigorosa de responsabilidades
- Servir como demonstração de habilidades backend para portfólio
- Permitir integração futura com microserviços

---

## Decisão

O sistema irá expor **endpoints REST** para cada módulo:

- Módulo de Viagem
- Módulo de Despesa
- Módulo de Checklist
- Módulo de Usuário

Os endpoints seguirão **URIs baseadas em recursos** e **semântica HTTP**:

- GET → leitura
- POST → criação
- PUT/PATCH → atualização
- DELETE → exclusão

Todos os endpoints serão documentados usando **OpenAPI / Swagger**.

---

## Justificativa

REST foi escolhido por:

- Simplicidade
- Convenções amplamente conhecidas
- Fácil integração com front-end ou futuros microserviços
- Prototipagem rápida

GraphQL ou gRPC foram considerados, mas rejeitados devido à complexidade desnecessária neste estágio.

---

## Consequências

- Forte separação dos adaptadores em relação à lógica de domínio
- Contratos claros de entrada/saída
- Pronto para documentação e testes de API
- Fácil de estender ou versionar futuramente
